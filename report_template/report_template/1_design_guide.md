## 回路概要

## 実装する通信仕様の概要
本回路で扱う RS-232/UART 通信は、1本の信号線にビットを時間順に並べて送る非同期シリアル通信である。クロック信号そのものは送受信線には流さず、送信側と受信側が同じビット幅の時間間隔を前提として、start bit をきっかけに各ビットを順番に解釈する。

今回のテストベンチでは、送信側、受信側、接続線の関係は以下のようになる。
- データを送る側はテストベンチ `tb_wbuart_loopback.v` 内の送信モジュール `u_tx` である。
- テストベンチは、送信したい 8bit データを `tx_data` レジスタにセットする。
- テストベンチは、`u_tx` の `i_wr` 入力を 1 クロックだけ `1` にして送信開始を指示する。
- `u_tx` は `i_wr` を受けると、`i_data` の値を内部に取り込み、`o_uart_tx` 出力へ1ビットずつシリアルデータとして流す。
- 通常のループバック試験では、`u_tx` の `o_uart_tx` 出力はテストベンチ内で `u_rx` の `i_uart_rx` 入力へ直結される。
- `u_rx` は `i_uart_rx` 入力に流れてきたフレームを RS-232/UART の形式に従って受信し、取り出した 8bit のデータ部分を `o_data` に書き込む。

送信される1フレームの形式は以下のとおりである。

```text
1 start bit + 8 data bits + 1 even parity bit + 1 stop bit
```

各ビットの役割は以下のとおりである。
- `start bit`: フレームの開始を示すビットで、受信側が「ここからデータが来る」と判断するために使う。
- `8 data bits`: 実際に送りたいデータ本体である。今回でいえば、テストベンチが `tx_data` にセットした値がこの部分に入る。
- `even parity bit`: データビット内の `1` の個数が偶数になるように付ける誤り検出用ビットである。受信側はこのビットを確認し、値が合わなければ `o_parity_err` を立てる。
- `stop bit`: フレームの終了を示すビットである。受信側はこのビットが正しい値でなければ `o_frame_err` を立てる。

つまり、`tx_data` に入っている値そのものがそのまま1本の線に出るのではなく、start bit、parity bit、stop bit を付けたフレームに変換されて送信される。`u_rx` はそのフレームから 8 data bits の部分だけをペイロードとして取り出し、正常に受信できた場合に `o_data`、`o_wr` を更新する。

## `txuart.v`
### 入力信号
- `i_clk`: システムクロック (100MHz)
- `i_reset`: 同期リセット (Highアクティブ)
- `i_setup`: 設定レジスタ (31ビット。Baudレート分周値やパリティの有無などを設定)
- `i_break`: ブレーク送信要求 (シリアル線を長くLowにする)
- `i_wr`: 送信開始要求パルス (1クロック幅)
- `i_data[7:0]`: 送信データ
- `i_cts_n`: ハードウェアフロー制御 (CTS_N。本シミュレーションでは1'b0に固定)

### 出力信号
- `o_uart_tx`: シリアル送信線
- `o_busy`: 送信中フラグ

### 内部レジスタ/パラメータ
- `state`: 送信状態を管理するステートマシン (`TXU_IDLE`, `TXU_BIT_ZERO`〜`TXU_BIT_SEVEN`, `TXU_PARITY`, `TXU_STOP`)
- `baud_counter`: UART 1bit 期間内のクロック数を数えるカウンタ
- `lcl_data`: 送信データを保持・シフトするレジスタ
- `calc_parity`: 送信用に計算したパリティビット

### シミュレーションログ出力
- リセットから待機状態への遷移
- 送信開始受付
- 各データ bit の送信
- パリティ bit の送信
- フレーム送信完了

## `rxuart.v`
### 入力信号
- `i_clk`: システムクロック (100MHz)
- `i_reset`: 同期リセット (Highアクティブ)
- `i_setup`: 設定レジスタ (31ビット。パリティやBaud設定)
- `i_uart_rx`: シリアル受信線

### 出力信号
- `o_data[7:0]`: 受信データ
- `o_wr`: 正常受信完了時に 1 クロックだけ立つ信号 (done)
- `o_break`: ブレーク検出フラグ
- `o_parity_err`: パリティエラー検出フラグ (自動的にクリアされる)
- `o_frame_err`: フレーミングエラー検出フラグ (自動的にクリアされる)

### テストベンチ内でエミュレートするステータス信号
`rxuart` コアは必要最小限の機能に絞られているため、上位レイヤやテストベンチ内で以下のステータスおよびエラー信号を管理・ラッチしています。
- `rx_data_valid`: `o_data` に未読の有効データが格納されていることを示す信号 (`data_read` でクリア)
- `rx_overrun_error`: 未読データ（`rx_data_valid=1`）がある状態で次の正常フレーム（`o_wr=1`）を受信した場合に立つエラー信号
- `tb_parity_err`: `o_parity_err` を検出し、次の `data_read` まで保持するためのラッチレジスタ
- `tb_frame_err`: `o_frame_err` を検出し、次の `data_read` まで保持するためのラッチレジスタ

### 主要ステータス信号とテスト内容
#### `rx_data_valid` のセットおよびクリア
意味:
- `u_rx` が正常に 1 フレーム受信し、`o_data` に有効なデータが入っていることを示す。
- `data_read` (読出し完了通知) を入力するとクリアされる。

テスト内容:
- `CASE1`
  - 正常受信後に `rx_data_valid=1` になることを確認する。
  - `pulse_data_read()` 後に `rx_data_valid=0` になることを確認する。
- `CASE4`
  - 1 回目受信後に `rx_data_valid=1` となることを確認する。
  - 2 回目受信後も未読のため `rx_data_valid=1` のままであることを確認する。
  - `pulse_data_read()` 後にクリアされることを確認する。

#### `tb_parity_err` の検出
意味:
- parity bit が、受信したデータから計算した期待値と一致しないときに立つエラー信号である。

テスト内容:
- `CASE2`
  - `inject_frame(8'h55, 1'b1, 1'b1)` により、意図的に誤った parity bit を注入する。
  - その結果 `tb_parity_err=1` となることを確認する。
  - あわせて `rx_done=0`、`rx_data_valid=0` であり、正常受信扱いしないことを確認する。

#### `tb_frame_err` の検出
意味:
- stop bit が本来 `1` で終わるべきところ、そうなっていない場合に立つエラー信号である。

テスト内容:
- `CASE3`
  - `inject_frame(8'h33, 1'b0, 1'b0)` により、stop bit を意図的に `0` にする。
  - その結果 `tb_frame_err=1` となることを確認する。
  - あわせて `rx_done=0`、`rx_data_valid=0` であり、正常受信扱いしないことを確認する。

#### `rx_overrun_error` の検出
意味:
- 前に受信したデータをまだ読み出していない状態で、新しい正常フレームを受信した場合に立つエラー信号である。

テスト内容:
- `CASE4`
  - まず `8'hA5` を正常受信し、`rx_data_valid=1` とする。
  - `data_read` を行わないまま、続けて `8'h3C` を受信させる。
  - このとき `rx_overrun_error=1` になることを確認する。
  - 最後に `pulse_data_read()` を入力し、`rx_overrun_error=0` に戻ることを確認する。

## `tb_wbuart_loopback.v`
### 目的
- 案件で要求される主要エラーを含む機能を一通り検証する
- 実行パスと入出力値をシミュレーションログに残す

### テストケース
- `CASE1`: 正常な loopback 送受信
- `CASE2`: パリティエラーの強制注入
- `CASE3`: フレーミングエラーの強制注入
- `CASE4`: 未読データを残したまま2フレーム受信し、オーバーランを確認
